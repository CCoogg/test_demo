# 执行规则问题的解决方案（solution.md）

## 问题概述
在用例执行中存在以下高频异常：
- 用例与实际环境不一致（数据/书籍/状态不匹配）。
- Agent 未发生真实操作（点击/滑动未生效）。
- Agent “臆造”结果（说出不存在的书籍/状态）。
- 大模型重复思考或陷入循环（同一描述反复输出）。

## 总体原则
- 证据优先：只允许基于“可见 UI 证据 + 实际操作回执”进行结论输出。
- 操作闭环：每个动作必须有可观测的 UI 变化或可校验回执，否则判定为未执行。
- 失败可接受：当条件不足或环境不匹配时，明确标记为失败/无法验证，并记录原因。

## 解决方案

### 1. 用例与实际不一致（数据缺失）
症状：用例要求“红楼梦/西游记”，但书架只有“111.txt”。

处理规则：
- 先执行“前置条件补齐”。能补齐则补齐；不能补齐则失败/无法验证。
- 前置条件补齐方式：
  - 通过导入流程创建需要的书籍。
  - 若文件缺失，允许使用“等价替代物”仅在用例明确允许的情况下；否则判定无法验证。
- 记录策略：
  - 在执行记录中写明“缺失的前置条件”和“补齐尝试”的具体步骤与结果。

### 2. 未发生真实操作（点击/滑动不生效）
症状：日志显示执行了点击/拖动，但界面无变化。

处理规则：
- 强制校验：每个动作后必须进行 UI 差异校验（如元素状态、数值变化、页面跳转）。
- 若 2 次连续动作无 UI 变化：
  - 尝试备用操作路径（例如点击元素中心点、长按、重新聚焦控件）。
  - 仍无变化则判定“执行失败”，停止该步骤后续动作并记录。
- 禁止“假成功”：无 UI 变化不得写通过。

### 3. 结果臆造（不存在的书籍/状态）
症状：Agent 输出大量不存在的书籍。

处理规则：
- 输出约束：所有“书籍名/数值/状态”必须来源于 UI 证据（截图/元素树）。
- 若未能在 UI 中检索到目标信息：
  - 不得输出猜测；直接标注“无法验证”。
  - 观察证据必须附上对应元素的 UI 快照或树信息。

### 4. 大模型重复思考或循环
症状：相同指令被反复输出，无法推进动作。

处理规则：
- 设置“动作重试上限”：同一动作最多重试 2 次。
- 设置“思考循环检测”：连续 2 次输出相同计划但无 UI 变化，判定为循环。
- 触发循环时：
  - 记录“循环原因”和最后一次可见证据。
  - 标记当前步骤失败并停止该用例执行，避免污染输出。

## 关键机制建议（不改代码，仅作为规则）
- 行动-证据绑定：每个 step 必须有一条观测证据（UI 树/元素快照/截图）。
- 失败优先判定：环境不一致或证据不足时，倾向失败/无法验证而非猜测通过。
- 证据采样最小化：只截取“被操作元素”的 UI 树片段，降低成本但保留可复核性。

## 输出与记录约束
- 仅输出符合 `agent_output.schema.json` 的结构化 JSON。
- Steps/Assertions 与 UI 信息严格隔离。
- Assertions 采用 suggestion 模型并附 confidence。

## 期望效果
- 避免“假执行”“假成功”和臆造结果。
- 用例失败可解释、可复核。
- 让 Registry/YAML 的后续产物质量可控。
