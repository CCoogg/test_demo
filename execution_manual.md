# 执行手册（不改代码版）

## 目标
- 确保 Agent 执行过程可复核、可解释、可控，避免“假执行/臆造/循环”。

## 核心原则
- 证据优先：任何结论必须有 UI 证据。
- 动作闭环：每个动作后必须观察到 UI 变化或可校验回执。
- 失败可接受：证据不足或环境不一致时，明确标记失败/无法验证。

## 执行流程（固定节奏）
1. 读用例
2. 前置条件检查
3. 数据适配
4. 执行步骤
5. 每步证据检查
6. 结果判定与记录

## 关键规则

### 1) 前置条件检查与数据适配
- 用例数据与现实不一致时，允许“适配现有资源”，但必须：
  - 在 UI 中能实际看到该资源
  - 在执行记录中写明“原始用例数据”与“适配后的实际资源”
- 不得凭空创造不存在的资源名。

### 2) 动作与证据闭环
- 每个动作后必须做 UI 变化验证：元素状态、数值变化、页面跳转等。
- 若 2 次连续动作无 UI 变化：
  - 尝试备用操作路径
  - 仍无变化则判定失败并停止该步骤

### 3) 禁止臆造
- 所有书名/数值/状态必须来自 UI 证据。
- UI 中无法验证的内容，直接标记“无法验证”。

### 4) 循环与重试
- 同一动作最多重试 2 次。
- 连续 2 次输出相同计划且无 UI 变化，判定为循环，停止该用例并记录原因。

## 证据最小集
- 只需保留“被执行动作对应元素”的 UI 树或元素快照。
- 如有关键数值变化，需记录变化前后对比证据。

## 结果记录要求
- 每条用例必须记录：
  - 前置条件是否满足
  - 数据适配情况（如有）
  - 关键证据点
  - 失败/无法验证原因（如有）

## 质量闸口（强制）
- 无证据不得判通过。
- 证据不足或环境不一致时，默认失败/无法验证。

## 执行前提醒（每次运行前必须宣读/确认）
- 任何结论必须有 UI 证据。
- 每个动作后必须观察到 UI 变化。
- 证据不足就失败/无法验证。

## 启动提示词模板（每次运行需粘贴）
执行前提醒（必须遵守）：
1. 任何结论必须有 UI 证据（截图/元素树/元素快照）。
2. 每个动作后必须观察到 UI 变化；连续两次无变化判定失败并停止该步骤。
3. 证据不足或环境不一致，默认失败/无法验证。

数据适配规则：
- 用例数据与现实不一致时，允许适配为“UI 中实际可见的资源”执行；
- 必须在执行记录中写明“原始用例数据”与“适配后的实际资源”；
- 禁止臆造不存在的资源名。
